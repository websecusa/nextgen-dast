Create a backup

Runs mariadb-dump against the live database and writes a gzipped .sql file into /data/backups/. Uses --single-transaction for a non-locking, point-in-time consistent snapshot. Triggers, routines and events are included.

For a multi-gigabyte database the dump can take a minute or more. The browser stays on this tab until it finishes — that is normal; do not close the tab.

Restore from a backup

Warning: a restore replays the SQL inside the uploaded dump against the current database. Any conflicting rows (same primary key) will fail individually and be reported in the restore message; tables that already exist with a different schema will produce errors. Take a backup first if you are not sure.

Maximum upload size: {{ '%.2f' % (max_restore_bytes / 1024 / 1024 / 1024) }} GB. Both .sql and .sql.gz are accepted; the server auto-detects gzip from the first two bytes.

Operational notes

  • The MariaDB container is configured with max_allowed_packet=1G, net_read_timeout=600 and innodb_log_file_size=256M so dumps and restores involving large raw_data blobs survive without packet-too-large errors.
  • Backups are stored under /data/backups/ on the host (mounted from ./data/backups). They survive container restarts and are included if you snapshot the host volume.
  • A backup taken on this version (2.1.1) is intended to be restored onto the same version. Restoring across major schema changes is not supported by this tool — use migrations instead.
